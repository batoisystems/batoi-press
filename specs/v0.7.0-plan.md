## Goal

Close the role-governance gap before `1.0.0` by enforcing the roles already stored in user records and documenting the first stable access model for admin routes.

## Release Theme

Role-aware admin access.

## Scope

### 1. Central Admin Access Policy

Status: Pending

Tasks:

- [x] Define route access rules in one reusable security helper.
- [x] Keep owners and admins on full access.
- [x] Allow editors to manage publishing routes: pages, posts, media, menus, and AIF assist.
- [x] Allow authors to manage post routes.
- [x] Limit viewers to the dashboard.

Acceptance:

- Authenticated users can only reach routes permitted by their role.
- The policy is easy to review and update without scanning every controller.

### 2. Blocked Access Handling

Status: Pending

Tasks:

- [x] Return a clear admin 403 page for blocked routes.

Acceptance:

- Unauthorized requests do not perform the underlying admin action.
- Audit review shows which user attempted which blocked route.

## Out of Scope for v0.7.0

- Per-content ownership checks for author-owned posts.
- Fine-grained field-level permissions.
- Role-specific dashboard personalization beyond navigation filtering.
- Breaking changes to user storage, content files, theme templates, public routes, or update package structure.

## Recommended First Implementation Slice

Start with a small `AdminAccess` helper and route-level enforcement in `Router`. Once server-side access is enforced, filter sidebar navigation to match the same policy.

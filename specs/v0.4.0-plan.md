## Goal

Harden Batoi Press for pre-1.0 production pilots by closing governance gaps, improving repeat admin workflows, and strengthening release/update confidence while keeping the runtime flat-file, PHP-only, database-free, and FTP-deployable.

## Release Theme

Operational hardening and governed administration.

## Scope

### 1. User Lifecycle Governance

Status: Complete

Tasks:

- [x] Add edit-user screen for email and role updates.
- [x] Add password reset form for existing users.
- [x] Add disable/reactivate actions with CSRF protection.
- [x] Prevent disabling the current user and the last owner account.
- [x] Prevent demoting the last owner account.
- [x] Exclude disabled users from successful authentication.
- [x] Audit user updates, password resets, disables, and reactivations.

Acceptance:

- Admins can maintain existing accounts without editing `users.json`.
- Disabled accounts cannot log in.
- At least one owner account remains active.
- User lifecycle actions are visible in the audit log.

## Out of Scope for v0.4.0

- Mandatory database storage.
- Mandatory Composer, Node.js, Git, or command-line runtime requirements.
- Mandatory Batoi Platform account or network service.
- Mandatory signed package enforcement.
- Breaking content, theme, installer, or update package contracts.

## Verification Plan

- Run `php -l` on changed PHP files.
- Run `php radpress/tests/smoke.php`.
- Run `php radpress/tests/update_runner.php`.
- Run `php tools/build-release.php`.
- Run `php tools/generate-release-manifest.php`.
- Browser-check `/admin/users`, user edit/reset flows, filtered list screens, and update surfaces at desktop and mobile widths.

## Recommended First Implementation Slice

Start with User Lifecycle Governance. It is a contained governance gap, it directly advances the 1.0 checklist, and it avoids changing content or update package contracts before the release safety tests are expanded.
